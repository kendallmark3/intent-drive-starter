Architecture / Intent-Driven Engineering / Rev. 01

Every change traces back to a written intent — nothing ships that can't be pointed at a reason.

Most AI-assisted workflows start from a prompt and hope the model inferred the right thing. This starter inverts that: the business outcome, the constraints, and the acceptance criteria get written down before planning starts, and every agent and hook downstream is wired to check its work against that document — not against a diff, and not against a guess.

The loop

  1. N01 Intent

    Project Intent

    intent/project-intent.md

    Purpose, business outcomes, and constraints for the whole system. Written once, revisited whenever the system's reason for existing changes.

  2. N02 Intent

    Current Feature

    intent/current-feature.md

    Outcome, requirements, constraints, and acceptance criteria for this specific unit of work. Every node below either produces from this or is checked against it.

  3. N03 Plan

    Planner Agent

    agents/planner.md · skills/feature-planner.md

    Breaks N02 into implementation steps. A step that doesn't trace back to the stated intent doesn't get planned.

  4. N04 Gate

    Pre-Implement Hook

    hooks/pre-implement.md

    Validates that intent is actually confirmed before a single line of code gets written. This is the check that stops "start coding and figure it out" cold.

  5. N05 Build

    Developer Agent

    agents/developer.md

    Implements against the plan and the shared context below — never against its own assumptions about how the system works.

    reads context/
  6. N06 Verify

    Post-Implement Hook + QA Agent

    hooks/post-implement.md · agents/qa.md

    Runs tests and updates docs before the change is allowed to be considered finished. Nothing skips this step to save time.

    reads context/
  7. N07 Gate

    Pre-PR Hook

    hooks/pre-pr.md

    Verifies checks are actually green before a pull request is allowed to exist — a second gate, not a rerun of the first.

  8. N08 Review

    Reviewer Agent

    agents/reviewer.md

    Confirms the change satisfies N02's acceptance criteria — not just that it compiles, passes CI, and looks reasonable in a diff.

    ↑ checked against N02 — Current Feature
  9. N09 Ship

    CI Pipeline

    .github/workflows/ci.yml

    The last automated gate before merge — by the time code reaches it, it has already been checked against intent twice.

Why teams start here

Traceability, not archaeology

Every diff points back to a written outcome and a set of acceptance criteria. No one has to reconstruct "why was this built" from commit messages six months later.

Validation happens before code, not after

The pre-implement hook physically blocks work from starting until intent is confirmed — the gate exists before the expensive part of the loop, not after it.

Context stays living, not stale

Architecture, business rules, and security constraints live in one place every agent reads from — so "the docs were out of date" stops being the excuse for a bad implementation.

One loop, whether it's agents or humans

The same nine steps apply whether an AI agent or a person is doing the work. Swapping in more automation later doesn't mean rebuilding the process.

Review checks intent, not just syntax

The reviewer's job is to verify the acceptance criteria were met — a change can pass every test and still fail review if it solved the wrong problem.
